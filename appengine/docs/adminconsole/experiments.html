Experiments

You use experiments to roll out features for your app slowly over a period of time, similar to what Google does when rolling out a feature over a few days or weeks. Experiments also allow you to do A/B Testing. Experiments work by splitting incoming requests to different versions of your app.

To set up experiments, you will need to choose a version, specify the amount of traffic it should receive, and choose the type of splitting to use. It also important to pay attention to the effect of caching.

  • Adding and Configuring Experiments for Your App
  • The Types of Splitting: IP Address and Cookie
  • How Caching Impacts Experiments

Adding and Configuring Experiments for Your App

You can find the settings for configuring your experiments on the Versions page in the Admin Console. When you add an experiment, you select the percentage of traffic to send to the experiment. Experiments can only be run on versions that aren't the default version. Make sure you have at least two major versions of your app available, so that you have one non-default version.

Traffic can be split either using an IP Address or a Cookie. This setting applies globally to all experiments run by your app.

The Types of Splitting: IP Address and Cookie

Each app can only have one method for splitting traffic, which will be used for all its experiments:

  • IP Address. This is the simplest to use, but has some significant limitations. Each incoming request has its IP address hashed to a value between 0 and 999. This is the value that is used for splitting traffic.
  • Cookie. Cookie splitting works by using HTTP cookies to assign users a value between 0 and 999. One advantage to this approach is that it is much easier to get a uniform distribution, and hence we allow smaller slices of traffic (as small as 0.1%). Cookies are also designed to be sticky.

Limitation of IP Address Splitting

IP address splitting has some significant limitations:

  • Non-Uniform Distribution. The way IP addresses are assigned is not random or uniform. Your app may see the majority of traffic from a relatively small number of IPs. It may also be the case that certain countries or languages map to a relatively small number of IPs. Large corporations typically use Network Address Translation (NAT) to map large numbers of users to single IP. In statistical terms, if you ask for "10%" of traffic, the mean is "10%" but the standard deviation is large. When asking for 10% of traffic, the standard deviation is about 2 or 3%. When asking for 1% of traffic, the standard deviation is about 1%. This is likely fine when launching new features, but is very problematic when doing A/B testing.
  • Only Slightly Sticky. IP addresses are reasonably sticky, but with some gotchas. Users connecting from cell phones may have a shifting IP address throughout a single session. Similarly, a user on a laptop may be moving from home to a cafe to work, and hence also shifting through IP addresses.
  • Internal Requests Don't Work. Requests sent to your app from outside of Google's cloud infrastructure will work normally. However, requests that are sent from your app to another app (or to itself) will not be sent to an experiment. The issue is that all these requests will originate from the same IP (or a smaller number of IPs). To avoid this problem, App Engine will simply not do any traffic splitting on requests sent from app to app within Google's cloud infrastructure.

Limitations of Cookie Splitting

There are some limitations with cookie splitting:

  • Cookie Management May Be Required. Browser-based apps work fine, as browsers automatically manage cookies. If you are writing a mobile app or a desktop client of some kind, then you will have to make your mobile app or desktop client manage cookies. This means that when App Engine sends back a 'Set-Cookie' header, the cookie will need to be stored and then sent again with each subsequent request.
  • Internal Requests Require Extra Work. Requests sent from your app server-side to another app (or to itself) can be sent into an experiment, but it requires extra work. It is possible to forward the user's cookie as part of the request. However, internal requests that don't originate from the user are not recommended for experiments.

Setting the Cookie

The cookie is automatically set for you. The cookie App Engine uses for traffic splitting is GOOGAPPUID, and its range of valid values is 0 to 999. This cookie is set to a random valid value if and only if the following are true:

  • Your app is configured to use cookie splitting.
  • The HTTP request from the user does not already contain the GOOGAPPUID header.
  • The response from your app does not already contain a Set-Cookie: GOOGAPPUID=... header.

However, if you disagree with the way in which the cookie is set, then you can override the behavior simply by sending a Set-Cookie: GOOGAPPUID=... header of your own. If App Engine sees a Set-Cookie: GOOGAPPUID=... header, then it will not attempt to overwrite or override it.

An Example

Each request arriving at an app is sent to a particular version of the app. Normally this is decided by the URL. For experiments, this can also be decided by the traffic splitting method and the experiments that are running.

Let's say your app is called codeninja, and has three active major versions: alpha, beta, and default (the default version). All traffic sent to http://codeninja.appspot.com will go to the default version. It's possible to go to the other versions by including their version name as a prefix. For example, http://beta.codeninja.appspot.com would go to the beta version.

Let's say you have two experiments running. You are rolling out beta slowly to all your users, and it is currently at 20% of all traffic. App Engine automatically assigns a range of values that will send an end user to beta. For example, maybe this range is 0 to 199 (20% of all available values). When an end user arrives at your app, App Engine will either hash the IP of the end user or examine their cookie (determined by the traffic splitting method). If the result of that is in the range 0 to 199, then the end user is sent to the beta version.

At the same time, you also have a highly experimental version (alpha) which is receiving 1% of traffic, to see how it influences user behavior. Under the covers, App Engine has assigned this version its own range of values guaranteed not to overlap with beta. For example, maybe this range is 990 to 999.

In this example, if the end user's cookie or hashed IP was in the range 200 to 989, then she would be sent to default. In the range 0 to 199, she'd be sent to beta. Finally, in the range 900 to 999, she'd be sent to alpha.

How Caching Impacts Experiments

When running experiments, caching is a big concern. An app is typically a combination of static and dynamic resources. Dynamic resources (pages that change with every request) should set "Cache-Control" or "Expires" headers to indicate this to proxies. These types of pages will behave normally. However, for more static content, it's important to pay careful attention.

Here are some topics to consider:

  • An Example: General Caching Concerns
  • Caching and Experiments
  • Caching and Vary: Cookie
  • Other Resources

An Example: General Caching Concerns

Let's say your app is called codeninja, and it's hosted on http://codeninja.appspot.com. The app has an awesome logo, which is hosted at http://codeninja.appspot.com/awesome.png. Since the logo does not change frequently, you set Cache-Control: public, max-age=86400. This means the logo can be cached by proxy servers, but only for 86400 seconds (or 1 day). Time comes to release a new version of codeninja. This version is called moreawesome. As part of this, you've included a new logo to represent the more awesome features on the web site.

The simplistic approach is to leave the logo with the same URL: http://codeninja.appspot.com/awesome.png. If you do this, then even though the users will be seeing the new dynamic content for the page, the logo may still be the old logo for as long as 24 hours. During those 24 hours, there may be browsers or proxy servers that still believe the content to be fresh.

The best way to fix this problem is to rename the resource. Caching is keyed by the full URL. So if the URL is different, then there is no cache entry to serve. In our example, we could rename the logo in the moreawesome version to be http://codeninja.appspot.com/moreawesome.png.

Caching and Experiments

It's important to change the URL for cacheable content that varies between experimental versions.

Caching issues exist for all apps, but experiments will make subtle flaws in caching much more apparent. In the above example, the worst thing that happens is that the logo looks wrong for some users for no more than 24 hours. After that, the issue magically resolves itself. If instead you had both the original version and the moreawesome version running concurrently (using experiments), the logo would look wrong for as long as the experiment is running. In fact, some proxy servers and browsers will cache one version of the logo, and some will cache the other. So a new error case arises: users still on the original version might be served the moreawesome logo.

The solution is still the same: change the URL. If each version of a static resource is serving from different URLs, then those resources can happily co-exist in proxy servers and browser caches.

Caching and Vary: Cookie

Vary: Cookie may save some development effort, but has significant limitations.

Cookie splitting opens up a new avenue for approaching the problem. If the app sets the Vary: Cookie header, then this means that the uniqueness of a resource is computed by combining the cookies and the URL for the request. This approach will increase the burden on cache servers. There are 1000 possible values of GOOGAPPUID, and hence 1000 possible entries for each URL for your app. Depending on the load on the proxies between your users and your app, this may decrease cache hit rate. Also note that for the 24 hours after adding a new batch of users to an experiment, they may still be getting the old logo. However, it does save the development effort of renaming static resources that are changing between versions.

The Vary: Cookie technique doesn't work in all circumstances. In general, if your app is using cookies for other purposes, then you must consider how this affects the burden on proxy servers. If codeninja had its own cookie that had 100 possible values, then the space of all possible cache entries becomes 100 * 1000 = 100,000. That's very big. In the worst case, there is a unique cookie for every user. Two common examples of this are Google Analytics (__utma) and SiteCatalyst (s_vi). In this case, every user gets a unique copy, which will degrade cache performance severely.

Other Resources

In the example above, the corresponding Date and Expires headers are omitted for brevity. These headers are important, as not all proxy servers properly support the HTTP/1.1 Cache-Control header. If you want more information on caching in general, try these web pages:
